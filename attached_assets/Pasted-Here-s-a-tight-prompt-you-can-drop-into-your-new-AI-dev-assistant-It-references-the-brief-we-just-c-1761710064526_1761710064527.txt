# Build Prompt: Jupiter Portfolio Full CSV Export (MV3)

**Primary spec:** See this doc first — it’s the source of truth with endpoint details, auth behavior, architecture, and pitfalls:
`jupiter_portfolio_export_rebuild_brief.md` (attached in this chat)

## Goal

Rebuild the **Jupiter Portfolio → Full CSV Export** Chrome extension so a non-technical user can export **all** activity rows (10k+) to CSV **without manual “Load more” clicking**, **without keeping the popup open**, and **without using deprecated MV2/webRequestBlocking**.

Keep the **original v3 UI/UX** (progress in popup, same look/feel) but fix the MV3 + request-capture issues described in the brief.

## Key Requirements (read the .md)

* **Auth**: Requests must include both `authorization: Bearer …` and `x-turnstile-token: …`. Use **credentials:"omit"**. Tokens are captured by wrapping page‐world `fetch` on the activity page.
* **Endpoint**: `GET https://portfolio-api-jup.sonar.watch/v1/transactions/fetch?address=…&limit=100&before=…`

  * **limit ≤ 100** (server hard cap).
  * Use the **last `signature`** as `before` cursor.
* **Data**: Dedupe by `signature|owner`. Flatten `balanceChanges` into rows with columns:

  ```
  signature,owner,isSigner,blockTime,platformId,serviceName,fees,success,ticker,address,preBalance,postBalance,balanceChange
  ```
* **Long-running**: Job must keep running if popup closes. Use **service worker** + optional **offscreen document** (MV3-compliant).
* **No MV2 APIs**: Do **not** use `webRequestBlocking`. Use a **content-script bridge** injected into **page world** to capture headers and to signal the background.
* **CORS**: We don’t send cookies; endpoint accepts wildcard CORS with headers only.

## Architecture (summarized)

* **manifest.json (MV3)**:

  * `permissions`: `downloads`, `storage`, `notifications`, `scripting`, `activeTab`, **optional** `offscreen`.
  * `host_permissions`: `https://jup.ag/*`, `https://*.jup.ag/*`, `https://portfolio-api-jup.sonar.watch/*`
  * `background.service_worker`: `background.js` (module).
  * `action.default_popup`: `popup.html`.
* **content.js**: Injects **page-world** `bridge.js` (via `chrome.scripting.executeScript` or a tiny DOM script tag).
* **bridge.js (page world)**: Wraps `window.fetch`, captures headers for the target endpoint, stores last valid pair, and `postMessage`s progress/diagnostics.
* **background.js**:

  * Listens for popup `export:start`.
  * Checks it has captured headers; if not, instructs user to click “Load more” once.
  * Paginates (`limit=100`, `before=lastSignature`), dedupes, streams to CSV in memory.
  * Emits progress via notifications + runtime messages.
  * Finalize CSV: either via **offscreen document** (File API) or **data URL** fallback with `chrome.downloads.download`.
* **popup.html/js**: Same original UI. Shows live progress and diagnostics. Can start/stop/resume jobs.

## Logging & Diagnostics (must-have)

* Popup “Diagnostics” button that requests from background:

  * `capturedCount`, `headersReady`, `lastStatus`, `hasBridge`, `pagesFetched`, `rowsInCsv`, `cursor`, `rateLimitBackoffs`.
* Background console logs with **timestamps** and **tabId**.
* Bridge console logs: `[bridge] armed`, `[bridge] captured headers`, `[bridge] progress …`.

## Error Handling

* Gracefully handle:

  * Missing headers ⇒ instruct user to click “Load more” once, then Start.
  * `429/5xx` ⇒ exponential backoff (≤ 7 attempts), jitter, then resume.
  * Network timeouts ⇒ retry with backoff.
  * Token rotation mid-run ⇒ re-arm capture and wait for next user “Load more” click (persist partial state and resume).

## Deliverables

1. **Load-unpacked-ready ZIP** with:

   * `manifest.json` (MV3, validated)
   * `popup.html`, `popup.js`, `popup.css`
   * `background.js`
   * `content.js`
   * `bridge.js`
   * `offscreen.html` (if used)
   * `/icons` (16/48/128)
   * `README.md` (install + usage + troubleshooting)
2. **Minimal code diffs** from original v3 except what’s required to:

   * Move to MV3
   * Replace webRequestBlocking with page-world bridge
   * Add long-running/export logic + diagnostics
3. **Acceptance tests** (documented in README):

   * On Jupiter activity page, click “Load more” once → Start export → popup may be closed → export completes → CSV downloads.
   * Works on wallets with **10k+** txs (no manual paging).
   * `limit=100` honored, pagination via `before` cursor.
   * CSV columns match spec and counts line up with site totals ± hidden/spam filters.
   * No MV2 warnings (`webRequestBlocking`) or CSP inline-script errors.

## Edge Cases to Simulate

* Very large histories (10k+).
* Intermittent `429` and `504`.
* Token rotation mid-run (requires another “Load more” click to refresh headers).

## Notes from Prior Failures (don’t repeat)

* Don’t rely on `webRequestBlocking` or MV2 keys.
* Don’t try to capture headers from extension world only — **must** instrument page-world `fetch`.
* Don’t exceed `limit=100` (server returns 400).
* Avoid `credentials: "include"` — CORS will block with wildcard ACAO.
* Ensure background can **run to completion without popup**; downloading must not depend on popup context.
